# Django MFA Audit Persistence Contracts

Status: accepted design gate
Date: 2026-06-15

## Decision

The next Django integration layer will persist local MFA audit events for HOTP
verification and HOTP resynchronization outcomes. Audit persistence must remain
separate from device counter state so `hotp.counter-advance`,
`hotp.replay-prevention`, and `hotp.resync-bounded` can keep their current
atomic update boundary.

Audit rows must be derived from `HOTPAuditRecord` and `HOTPResyncAuditRecord`.
They must not store submitted OTPs, raw MFA seeds, provisioning URIs, Fernet
keys, recovery codes, passwords, production credentials, or raw encrypted
secret values.

## Model Shape

The implementation should add one package-owned audit event model unless a
later implementation task finds a concrete reason to split verification and
resynchronization into separate tables.

Each audit event should include:

- `user`, nullable only when the related user has been deleted;
- `device`, referencing `HOTPDevice` and nullable only when the device has been
  deleted;
- `factor`, fixed to `hotp` for this milestone;
- `event_type`, either `verification` or `resynchronization`;
- `submitted_outcome`, copied from the audit record as `accepted` or
  `rejected`;
- `result_classification`, copied from the audit record;
- `server_counter`;
- `matched_counter`, nullable when no counter matched;
- `next_counter`;
- `look_ahead`, nullable for resynchronization events;
- `search_window`, nullable for verification events;
- `replay_window`;
- `submitted_count`, nullable for verification events;
- `attempted_at`, copied from the audit record;
- `created_at`, generated by the database model.

The model may include indexes for common local investigations, such as:

- `user`, `attempted_at`;
- `device`, `attempted_at`;
- `event_type`, `result_classification`;
- `attempted_at`.

## Classification Coverage

Audit persistence must cover every current HOTP service classification:

- verification `success`;
- verification `counter_window_match`;
- verification `invalid`;
- verification `replay`;
- verification `throttled`;
- resynchronization `resync_success`;
- resynchronization `invalid`;
- resynchronization `replay`;
- resynchronization `excessive_drift`;
- resynchronization `throttled`.

The persisted classification value should match the service result exactly so
operators and verification helpers can reason about the same vocabulary used by
`django_mfa_toolkit.hotp`.

## Retention and Deletion

Audit events are local security records. The package should preserve audit
events when devices are deactivated, and should not require audit deletion for
normal device removal. Foreign keys may use `SET_NULL` for user and device
references so historical outcomes remain available after account or device
cleanup.

The package should not define a default automatic deletion policy in the model.
Applications can layer their own retention policy based on compliance needs.
Documentation should describe retention as an application decision.

## Transaction Boundary

Audit persistence should be available to the model-backed adapters without
weakening counter safety:

1. Lock the HOTP device row inside `transaction.atomic()`.
2. Run the HOTP verification or resynchronization service.
3. Persist `hotp_counter` only when the service result is accepted.
4. Persist the audit event from the service audit record.
5. Return the existing service result type.

If audit persistence is optional, the default behavior must remain
backward-compatible and must not change return values. The implementation task
must define whether audit persistence failure should roll back the counter
update or fail open. The security-preferred default is to keep audit and counter
state in the same transaction when audit persistence has been explicitly
requested.

## Implementation Tests

The implementation task must add focused tests proving:

- verification audit events persist `success`, `counter_window_match`,
  `invalid`, `replay`, and `throttled` classifications;
- resynchronization audit events persist `resync_success`, `invalid`, `replay`,
  `excessive_drift`, and `throttled` classifications;
- persisted rows map every field listed in this contract;
- submitted OTPs, raw seeds, provisioning URIs, Fernet keys, recovery codes,
  passwords, production credentials, and `persisted_secret` values are absent
  from audit rows;
- failed, replayed, throttled, and excessive-drift attempts do not advance
  `hotp_counter`;
- audit event migrations are importable;
- audit persistence happens in the same transaction as counter updates when
  explicitly enabled.

## Validation Mapping

This design preserves and extends these controls:

- `hotp.audit`: audit outcomes move from in-process dataclasses to optional
  local persistence.
- `hotp.counter-advance`: persisted audit events must not cause counter updates
  on failed attempts.
- `hotp.replay-prevention`: replay outcomes are auditable without accepting the
  replayed code.
- `hotp.resync-bounded`: excessive-drift outcomes are auditable without
  widening the search window.
- `secret-storage.encrypted-at-rest`: audit rows must not store raw secrets or
  encrypted secret values.
